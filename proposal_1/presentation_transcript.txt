SLIDE 2 — INTRODUCTION AND BACKGROUND
================================================================
  • Businesses and organisations handle large volumes of documents
    daily — invoices, forms, contracts, and records

  • Extracting structured data from these documents is essential
    for operations, reporting, and compliance

  • Traditionally, this was done manually or with rule-based tools;
    AI-powered extraction is now an emerging alternative

  • Two categories of AI models are used for this purpose:

      Cloud-Based LLMs (e.g., GPT-4o, Gemini)
        — hosted remotely, accessed via paid APIs

      On-Device Small Language Models / SLMs (e.g., Qwen, Gemma,
        Phi, DeepSeek) — run locally on the organisation's hardware


================================================================
SLIDE 3 — PROBLEM STATEMENT (Part 1 of 2)
================================================================
THE PROBLEM:
  • Organizations process thousands of documents daily
    (invoices, forms, records, receipts)
  • Manual data entry is slow, expensive, and error-prone
  • AI can automate this — but which kind, and at what cost?

CURRENT DOMINANT SOLUTION — Cloud-Based AI (OpenAI, Google):
  • Charges per token → costs grow rapidly at scale
  • Sensitive documents sent to foreign servers
    → serious privacy and regulatory risk
  • Not viable for resource-constrained or data-sensitive
    environments


================================================================
SLIDE 4 — PROBLEM STATEMENT (Part 2 of 2)
================================================================
THE EMERGING ALTERNATIVE — On-Device Small Language Models (SLMs):
  • Run locally on a regular laptop via Ollama (free, open-source)
  • Fully offline — data never leaves the organization
  • No per-token cost after setup

BUT THE QUESTION REMAINS:
  • Are they accurate enough for real document extraction?
  • How do they compare in speed to cloud APIs?
  • When do they become cheaper than cloud at scale?

RESEARCH GAP:
  No existing study has simultaneously compared multiple local
  SLMs vs. cloud LLMs on the same task covering accuracy,
  latency, prompt sensitivity, and per-document cost.

SLIDE 7 — SCOPE & RESEARCH QUESTIONS
================================================================
SCOPE:
  • Document type:         Invoices and structured forms (synthetic)
  • Sample size:           500 documents (statistically justified)
  • Models tested:         3 local SLMs  vs.  2 cloud LLM APIs
  • Instruction variants:  3 prompt styles per model

RESEARCH QUESTIONS:
  Q1: Is there a meaningful accuracy difference between local and
      cloud AI — and does it vary with instruction wording?

  Q2: How much faster or slower are local models compared to
      cloud API services?

  Q3: At what daily document volume does a local model become
      cheaper than a cloud subscription?


================================================================
SLIDE 8 — RESEARCH OBJECTIVES
================================================================
  1. Measure and compare extraction accuracy of local SLMs vs.
     cloud LLMs using the ANLS* metric.

  2. Compare end-to-end response latency (local vs. cloud).

  3. Build a reproducible per-document cost model for both
     local (hardware + electricity) and cloud (token billing)
     deployments.

  4. Identify the daily document volume at which a local
     deployment becomes cheaper than a cloud subscription.

  5. Analyse how instruction wording (prompt style) affects
     accuracy across all tested systems.

SLIDE 11 — RESEARCH METHODOLOGY (Part 1 of 2)
================================================================
EXPERIMENTAL DESIGN:
  • Controlled, quantitative comparison study
  • Every model receives identical documents and instructions
  • Results measured against a fixed, pre-built answer key

PHASE 1 — Setup (Weeks 1–6):
  • Literature review + model & hardware selection
  • Generate 500 synthetic test documents + answer key
  • Configure Ollama with all local models
  • Set up energy logging and cloud API access

PHASE 2 — Data Collection (Weeks 7–13):
  • Run each document through all 7 models
  • Apply 3 instruction prompt variants per model:
      A — Short:    "Extract vendor name, total, and date."
      B — Detailed: Full field-by-field natural language prompt
      C — JSON:     "Return only a JSON object with keys…"
  • Log: raw output, latency (ms), energy consumption (W)
  • Cloud: log token usage and billing records


================================================================
SLIDE 12 — RESEARCH METHODOLOGY (Part 2 of 2)
================================================================
PHASE 3 — Analysis (Weeks 14–18):
  • Compute ANLS* and BERTScore per model per prompt variant
  • Select best prompt per model for final comparison
  • Calculate per-document cost:
      Cloud → tokens × price per million
      Local → electricity + hardware amortisation
  • Determine breakeven volume (local vs. cloud)
  • Statistical significance testing on accuracy differences

SAMPLE SIZE JUSTIFICATION:
  Formula:  n = (Z² × p × (1−p)) / E²
  Values:   p = 0.80 | E = ±0.035 | Z = 1.96
  Result:   n ≈ 500 documents

  → Ensures a 4% accuracy difference is statistically real,
    not sampling noise, at 95% confidence level.
