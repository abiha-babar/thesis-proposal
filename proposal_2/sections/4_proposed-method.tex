\section{Proposed Method}
\label{sec:method}

\subsection{Overview of the Experiment}

This study will run the same set of tasks under two different setups and compare the results. In the first setup (Condition~A), the agent is given all available tools at once as a flat list in its instructions --- the standard way most agents work today. In the second setup (Condition~B), the agent uses the OpenClaw \cite{openclaw_2025} skill system, where it only sees tool names first and loads full instructions for a tool when it needs them. Everything else stays the same across both setups: the same tasks, the same models, and the same scoring method. This way, any difference in the results can only be explained by the change in how tools are loaded.

\subsection{Tools and Software Used}

No heavy frameworks are needed for this study. The full setup uses four things:

\begin{itemize}
    \item \textbf{OpenClaw} \cite{openclaw_2025}: The autonomous agent framework used for Condition~B. OpenClaw already implements progressive disclosure natively through its \texttt{SKILL.md} file system. It also generates detailed JSONL log files automatically, which Python scripts will read to extract all measurements.
    \item \textbf{Ollama} \cite{ollama_2024}: A free, open-source tool that downloads and runs AI models locally on a laptop. It handles all setup automatically and provides a simple web API that both Condition~A and Condition~B can call. Two local models will be tested: \texttt{gemma4:4b} by Google DeepMind \cite{gemma4_2026} and \texttt{qwen3.5:4b} by Alibaba \cite{qwen3_2025}.
    \item \textbf{Gemini 3.1 Pro} \cite{google_gemini_2025}: The primary cloud-based model, accessed through the Google Generative AI Python SDK. This serves as the upper-bound comparison and shows whether progressive disclosure helps equally on large cloud models.
    \item \textbf{GPT-4o} \cite{openai_gpt4o_2024}: A secondary cloud model accessed through the OpenAI Python SDK. Included to check whether findings are consistent across different cloud providers.
    \item \textbf{Python}: Used to dispatch tasks, read the JSONL log files that OpenClaw produces, and compute all metrics. No additional agent framework is needed.
\end{itemize}

\subsection{The Two Agent Setups}

\subsubsection*{Condition A --- Flat Tool Use (Baseline)}

In Condition~A, all 20 available skills are converted into a flat JSON schema list and added to the agent's system prompt at the start of every task. Each entry includes the skill name, a one-sentence description, and the expected input and output types. The agent reads through this full list before every reasoning step and picks the tool it thinks it needs. This matches the standard function-calling approach used by most AI frameworks today.

\subsubsection*{Condition B --- Progressive Skill Disclosure (Experimental)}

In Condition~B, the agent uses OpenClaw's \cite{openclaw_2025} native \texttt{SKILL.md} architecture. The system prompt contains only a short index: the names and one-line descriptions of all 20 skills. This takes up far fewer tokens than a full schema list. When the agent decides it needs a specific skill, OpenClaw loads the full \texttt{SKILL.md} file for that skill into the active context. Once the skill's task is finished, OpenClaw removes those instructions before the next reasoning step begins. Python scripts read the JSONL log files that OpenClaw writes to measure exactly when each skill was loaded, how many tokens were added, and how long each step took.

\subsection{Skill Library}

A library of 20 skills will be created for this study, stored as \texttt{SKILL.md} files. The skills are drawn from three real-life domains that OpenClaw \cite{openclaw_2025} is specifically designed to handle, and that represent tasks a real user would give to a personal autonomous agent on a daily basis. Using real-life domains rather than synthetic tasks makes the results more meaningful and easier for others to relate to.

\begin{enumerate}
    \item \textbf{Communication (6 skills):} Drafting an email reply given a thread summary; composing a meeting invitation with specified details; summarising a set of unread messages; writing a short Telegram or WhatsApp message; generating a weekly update for a contact; and formatting a formal written response from bullet-point notes.

    \item \textbf{Research and Information (8 skills):} Searching the web for a specific topic and returning a structured summary; summarising a given article or web page; comparing two products, services, or options side by side; verifying whether a specific factual claim is supported by online sources; monitoring a keyword and reporting new results; generating a structured research briefing from multiple sources; extracting key statistics from a retrieved page; and ranking a list of options based on specified criteria.

    \item \textbf{Task and Planning (6 skills):} Creating a prioritised to-do list from a description of goals; setting a reminder with a natural-language time expression; generating a daily morning briefing from a set of inputs; tracking progress on a multi-step goal and reporting what is done and what remains; writing a short project status summary; and producing a formatted action-item list from meeting notes.
\end{enumerate}

Every \texttt{SKILL.md} file contains three sections: (1) \textbf{Applicability Conditions} --- when the skill should be used; (2) \textbf{Execution Policy} --- a numbered list of steps the agent must follow; and (3) \textbf{Termination Criteria} --- when the skill is done or should stop. In Condition~A, the same 20 skills are represented as flat JSON schema blocks loaded into the system prompt all at once. Two library sizes will be tested --- 10 skills and 20 skills --- to see whether the performance gap between the two conditions grows as the skill library gets larger.

\subsection{Tasks}

A set of 40 tasks will be used, divided into three levels of difficulty:

\begin{itemize}
    \item \textbf{Level 1 --- Single-skill tasks (15 tasks):} The agent needs exactly one skill to complete the task. Example: \textit{``Extract the invoice number and total amount from this PDF.''}
    \item \textbf{Level 2 --- Two-skill tasks (15 tasks):} The agent needs to use two skills in order. Example: \textit{``Extract the supplier email from this form and save it to a CSV file.''}
    \item \textbf{Level 3 --- Multi-skill tasks (10 tasks):} The agent needs three or more skills, with at least one decision point in the middle. Example: \textit{``Search for the most recent contact details for the company named in this document and write a summary report.''}
\end{itemize}

Every task is run under both conditions and with all four model backends. Each task is repeated three times to allow averages and standard deviations to be reported. This gives a total of $40 \times 2 \times 4 \times 3 = 960$ individual agent runs.

\subsection{Measures}

\begin{itemize}
    \item \textbf{Task Completion Rate:} Was the final answer correct? Text extraction tasks will be scored using the ANLS* metric \cite{anlsstar_2024}, which gives partial credit for nearly-correct answers. File and web tasks will use a simple pass or fail score.

    \item \textbf{Hallucinated Skill Calls:} How many times did the agent try to call a skill that does not exist, or call a real skill in a situation that violates its applicability conditions? This is recorded directly from the OpenClaw JSONL logs.

    \item \textbf{Trajectory Compliance Score:} Did the agent follow the numbered steps in the skill's execution policy in the correct order? Each task run is scored by comparing the log of agent actions against the ground-truth policy for the relevant skill.

    \item \textbf{Prompt Token Count:} How many tokens were in the agent's input at each reasoning step? For local models, this comes from the Ollama API response. For cloud models, it comes from the API response metadata.

    \item \textbf{Latency:} How long did each task take from start to finish, measured in seconds and averaged across all three repetitions.

    \item \textbf{Inference Cost:} For cloud models, cost is taken from the API billing records (tokens used multiplied by the published price per million tokens). For local models, cost is calculated from power consumption measured with \texttt{powermetrics} on Mac or \texttt{powertop} on Linux \cite{tokenpowerbench_2024}, multiplied by the local electricity rate.
\end{itemize}
